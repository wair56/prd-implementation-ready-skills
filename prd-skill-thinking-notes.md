# PRD Skill 优化思考沉淀

本文档整理这一路优化 `prd-implementation-ready` skill 时沉淀下来的例子、判断方式和可复用方法。核心目标不是把 PRD 写得更长，而是让 AI 在和用户讨论业务时更像一个真正的产品经理：先理解业务，再发现流程，再落成可实现的需求文档。

## 1. 总原则

这套 skill 的主线应该是：

```text
用户上下文
-> 深度业务理解
-> 业务观察视角发现小流程
-> 主流程与并行流程推演
-> 对象 / 状态 / 数据 / 页面 / 异常 / 反向路径
-> 可实现、可测试、可阅读的 PRD
```

不要一上来套模板。模板只能组织已经理解清楚的业务，不能代替业务理解。

## 2. 上下文优先：先吃透用户给的信息

用户给的材料、例子、纠错、旧文档、生成文档、业务口头说明，都是第一真源。AI 应先做 `context digest`，再进入 PRD 流程。

上下文消化需要提炼：

| 项目 | 要提炼什么 |
|---|---|
| 业务事实 | 用户明确说了什么，业务真实怎么跑 |
| 角色 | 谁操作、谁审批、谁付款/收款、谁查看、外部系统是谁 |
| 对象 | 单据、账户、资源、页面、任务、流水、来源明细 |
| 约束 | MVP 边界、既有系统行为、用户确认过的铁律 |
| 冲突 | 用户描述、旧文档、生成文档之间哪里不一致 |
| 缺口 | 不补就无法实现或无法测试的问题 |
| 推断 | AI 基于上下文做出的建议，必须标注为推荐或待确认 |

反例：一上来按“模块说明、主流程、页面、字段、异常”写，看起来完整，但业务事实可能已经偏了。

推荐做法：先说“基于你给的上下文，我理解这个业务是这样”，再进入流程、页面、字段。

## 3. 深度业务理解阶段

业务理解阶段不是开场白，而是正式的分析阶段。进入模块写作前，需要形成一个 `Deep Business Understanding Package`。

这个包至少包含：

| 部分 | 要做什么 |
|---|---|
| 上下文消化 | 业务事实、角色、对象、约束、冲突、缺口、推断 |
| 外部调研 | 看同领域/相邻领域成熟产品、SaaS、ERP、CRM、审批、结算、后台、合作方门户等做法 |
| 流程别扭分析 | 找重复录入、线下协调、校验太晚、对象职责混乱、无人负责失败恢复等问题 |
| 小流程发现 | 找隐藏在主流程旁边的操作流、通知流、回写流、恢复流 |
| 异常/反向理解 | 失败、拒绝、取消、退款、作废、红冲、超时、重试、部分成功、回滚 |
| 决策摘要 | 已确认锚点、推荐默认值、高风险待确认、边界和不做项 |

如果用户要求快，也可以做一个压缩版，但不能跳过。

## 4. 外部调研不是装饰

外部调研的目的不是堆链接，而是帮助判断业务形态、对象拆分、状态设计、页面组织和异常处理是否合理。

可调研的方向：

- 同领域成熟产品。
- 相邻领域的 SaaS、ERP、CRM、Marketplace、审批系统、账务系统、后台管理系统。
- 开源项目里的对象模型、审批流、账单流、发票流、任务流、后台 UI。
- 项目已有的旧 PRD、设计稿、接口文档、截图、代码和数据库结构。

使用调研结果时要标注来源和性质：

- 用户确认：可以写成确定规则。
- 现有系统行为：可以作为当前约束。
- 通用产品实践：写成推荐默认值。
- 无法确认：标为待确认。

## 5. 流程别扭分析

AI 不应该只复述用户给的流程，还要判断流程是否别扭。

重点看：

| 别扭点 | 典型表现 |
|---|---|
| 重复录入 | 用户在多个地方填同一信息 |
| 线下协调 | 需要微信、电话、表格才能完成系统内任务 |
| 校验太晚 | 到支付、结算、对账时才发现基础数据不对 |
| 一个对象职责过多 | 一个状态同时表示审核、支付、发票、推送、异常 |
| 无明确责任人 | 失败后没人知道谁处理、在哪里处理 |
| 过度人工审核 | 明明是确定性规则，却强行走人工审批 |
| 无恢复路径 | 异常只写“人工处理”，没有页面、字段、重算和回流 |

例子：

- 供应商消耗同步时车牌匹配不到车辆，不能只写“标异常”，还要有异常工作台、人工补全字段、补全后的校验、重新归集、审计记录。
- 支付成功后触发供应商输出工资明细，如果输出失败，支付状态不能回滚，但附加动作需要可重试、可见、可恢复。

## 6. 业务观察视角：发现稀碎的小流程

真实产品不是只有主流程。很多关键能力藏在小流程里。

建议用这些视角扫业务：

| 视角 | 发现什么 |
---|---|
| 角色视角 | 每个角色看什么、做什么、收到什么通知、如何恢复 |
| 对象生命周期视角 | 对象如何创建、锁定、失败、重开、作废、归档 |
| 触发事件视角 | 点击、导入、同步、回调、定时、超时、禁用、纠错 |
| 资源/价值视角 | 钱、额度、库存、权限、容量、可见性如何流转 |
| 数据来源视角 | 权威数据、复制字段、生成字段、迟到数据、回写 |
| 页面任务视角 | 用户在哪里发现、判断、操作、下钻、恢复 |
| 异常恢复视角 | 失败、拒绝、缺映射、重复、部分成功、回滚 |
| 租户/组织视角 | 同一主体跨租户、组织、项目、合同、角色时怎么处理 |

小流程发现表：

| 小流程 | 为什么重要 | 归属对象 | 触发 | 操作人/系统 | 入口/页面 | 数据来源 | 状态/规则影响 | 下游影响 | 异常恢复 | 范围决策 |
|---|---|---|---|---|---|---|---|---|---|---|

不是每个小流程都要变模块，但每个纳入本期的小流程都要有归属、触发、入口、数据、下游影响、异常和验收结果。

## 7. 进度控制字段闭环

例子：司机列表上有“上次工资发放时间/月份”。发起工资单时，系统自动取这个字段的下一个自然月作为工资单期间，且不可改。工资单真正支付/发放完成后，再回写这个字段。

这个字段不是展示字段，而是业务控制字段。

需要定义：

| 项目 | 要说明什么 |
|---|---|
| 字段含义 | 它代表业务进度处理到哪里 |
| 展示位置 | 列表、详情、卡片、筛选、导出 |
| 读取动作 | 哪个创建/操作流程读取它 |
| 默认推导 | 如何得到下一期，比如上次发放月 + 1 |
| 是否可改 | 默认锁定还是允许例外 |
| 连续性规则 | 是否允许跳期、重期、未来期 |
| 防重复 | 唯一键和幂等键 |
| 推进事件 | 创建、审核、支付成功、发放成功，哪个才推进 |
| 不推进事件 | 草稿、拒绝、取消、支付失败、支付中 |
| 反向处理 | 作废、重发、冲正、人工修正时字段如何处理 |

通用化后，不只适用于工资：

- 已结算至月份。
- 已开票至期间。
- 已处理至批次。
- 下次可创建期间。
- 上次发放/发券/发货/结算时间。
- API 额度已重置至日期。

## 8. 页面任务闭环

很多 PRD 的页面问题不是字段少，而是页面不知道为什么存在。

每个页面先回答：

| 问题 | 说明 |
|---|---|
| 页面为什么存在 | 它解决什么业务任务 |
| 用户先看什么 | 首屏默认视图、卡片、Tab、筛选 |
| 用户要判断什么 | 可用余额、待支付、异常、风险、可操作性 |
| 用户在哪里操作 | 按钮、抽屉、弹窗、详情页、批量栏 |
| 操作前看什么上下文 | 金额、来源、对方、状态、影响范围 |
| 成功后怎么反馈 | 状态、流水、通知、跳转、日志 |
| 失败后怎么恢复 | 失败原因、重试、补录、作废、人工处理入口 |

这一路提到的页面例子：

- 账户不应只有流水列表，还要有余额卡、钱包总额、按货主聚合余额、分货主下钻。
- 充值、提现、绑卡、联系人、手机号变更，都是账户操作，需要入口、校验、二次验证、结果反馈和审计。
- 资金明细要有回单、支付批次号、支付流水号、付款方/收款方、发起时间、完成时间。
- 供应商列表要考虑税号、费率、支付方式、合作情况、筛选项。
- 劳务供应商需要“查看关联司机”抽屉。
- 供应商禁用要通知相关司机或责任人，并说明待处理任务。
- 充值确认可以有验证码，属于高风险短操作的验证弹窗。
- 合并支付不是一个字段，而是一个选择、校验、合并、支付、回写的小流程。
- 对账单如果业务需要，应保留时分秒精度。
- 发票管理要校验税率、不含税金额、税额、价税合计。

## 9. 资金/资源流不是财务专用

我们反复遇到的问题，本质是资源闭环问题，不只是钱。

任何受控资源都要回答：

| 阶段 | 问题 |
|---|---|
| 来源 | 从哪里来，谁拥有，权威锚点是什么 |
| 可用 | 什么条件下可用，哪些状态不可用 |
| 占用 | 何时占用，是否冻结，是否影响可用量 |
| 组合 | 部分不足时能否多个来源组合 |
| 消耗 | 哪个事件真正消耗资源 |
| 释放 | 取消、失败、退款、作废时怎么释放 |
| 追踪 | 来源追踪是审计用途，还是控制释放 |
| 角色视图 | 不同角色看到哪些余额、原因和历史 |

非财务例子：

- 库存：下单占用，取消释放，支付成功扣减。
- 优惠券：领取、锁定、核销、退回。
- 预约名额：预约占用，超时释放。
- API 额度：分配、占用、消耗、重置。
- 权限/可见性：授予、撤销、历史记录可见。

## 10. 运营扶持金与资金流例子

这一路对运营扶持金的纠偏非常典型。

关键思考：

- 不要随便用“来账时点”做归集锚点。归属月份必须由业务权威字段决定。
- 可用余额和运营扶持金不应强制二选一，部分不足时可能需要组合使用。
- 来源池和来源追踪要分开。当前池子控制可用量，来源追踪解释历史，不一定一一对应释放。
- 货主资金和平台授予扶持金的关闭路径不同，不能都叫“退还”。
- 平台授予的扶持金，额度足够时可以确定性无审核扣减，生成已完成的账务/支付调整。
- 货主资金通常通过退款、核销、对账、结算释放，不应被“退还扶持金”直接释放，除非业务确认。
- 冻结资金如果用于保障后续对账/分润，不能简单写“不参与对账”。它应该在分润、归还扶持金、承包人可提现利润之间闭环。
- 平台端要能看到承包人资金、货主资金、授信占用、冻结、释放、异常锁定，而不是只有承包人自己的余额。

## 11. 能源充值/供应商预充值例子

用户纠正过一个关键点：能源充值不是内部预存登记。

正确理解：

- 能源充值是经营支付的一种。
- 对能源类供应商，可能是供应商预充值/预付款。
- 充值后形成供应商侧可用额度。
- 承包人或平台可以把额度分配给车辆。
- 后续车辆消耗从预充值额度中扣减。
- 需要供应商余额、分配明细、车辆使用、供应商对账、发票和支付轨迹。

因此 PRD 不能只写“能源账户余额 += R”，还要说清楚真实外部付款、供应商预付款、后续分配、消耗、对账和退款/冲正。

## 12. 供应商对账与发票例子

供应商对账不能只写消耗同步和对账总额。

需要分供应商类型：

| 类型 | 要处理什么 |
|---|---|
| 预充值供应商 | 预付款、余额、消耗扣减、车辆/订单分配、退款/冲正 |
| 非预充值供应商 | 应付生成、对账单、发票、支付申请、支付状态、失败重试 |
| 混合供应商 | 同一供应商不同产品/费用类型可能走不同账 |

发票也不是一行“发票登记”：

- 发票是子对象。
- 要有发票号唯一性。
- 要有税率、不含税金额、税额、价税合计。
- 要和对账单/支付金额做一致性校验。
- 要有上传、编辑、作废、红冲、附件、状态、入口、回写。

## 13. 成本、保险、司机工资例子

成本进入账单不能只看“费用存在”。

需要成本纳入门槛：

| 维度 | 说明 |
|---|---|
| 审核状态 | 是否审核通过 |
| 支付状态 | 是否已支付或无需外部支付 |
| 成本确认 | 是否已确认归属期 |
| 锁定期间 | 成本月是否已锁定 |
| 异常数据 | 缺车辆、缺供应商、缺合同怎么补 |
| 迟到数据 | 锁定后数据怎么处理 |

例子：

- 车辆费用未审核通过，不允许确认成本。
- 如果业务要求已支付才进成本，PRD 必须明确“审核通过 + 已支付”。
- 保险是年付，但成本可能要按月摊销。要区分现金支付和成本确认，生成摊销计划。
- 缺车辆的数据不能只“进入异常”，要有页面补全、校验、重算、回流。

司机工资公式也不能只写：

```text
max(保底, 按运单核算司机运费) + 加减项
```

必须问：

- 保底是什么保底：月保底、日保底、线路保底、车辆保底、合同最低？
- 按运单核算用哪个字段：司机运费、货主运费、承包人运费？
- 比较范围：同司机、同月、同车辆、同合同吗？
- 加减项是在比较前还是比较后？
- 新增司机当月没有金额，是否生成 0 元工资行？
- 工资单已支付后，是否才进入成本？
- 支付完成后是否回写上次劳务费发放时间和月份？

## 14. 货主对账、按月包车和按量计费

对账金额不能脱离商业对象。

例子：

- 按月包车通常要按车来，因为商业协议可能是车辆维度。
- 按月包车也可能有加减项、车辆数、活跃天数、换车、停运、调整。
- 按运输量应取运单上的货主运费或已确认应收额，而不是随便用重量乘单价。
- 对账单抬头应体现平台名称，因为本质上是平台向货主收钱。
- 展示上，按月包车要展示车辆数/车辆明细；按量计费要展示单数/运单金额。
- 同一业务时段重复对账不能简单按“货主 + 月份”阻断，因为可能存在并行项目、合同、线路、部分先对账。

## 15. 分润与平台服务费例子

分润不是一个公式，而是资金瀑布。

需要区分：

- 货主已收/已核销资金。
- 货主冻结资金。
- 平台授予扶持金。
- 承包人自有资金垫付。
- 平台服务费。
- 承包人可提现利润。
- 不足时继续抵扣。
- 0 或负数是否也生成分润单用于闭环。

用户明确指出：平台服务费要进平台口袋。  
所以 PRD 必须说明服务费付款方、收款方、费率快照、计费基数、扣收时机、账务影响。

分润作废也不能只写“释放占用”，要说：

- 对账单占用是否释放。
- 平台服务费是否冲回。
- 扶持金还款是否回滚。
- 承包人自有资金恢复如何处理。
- 已支付的分润款走作废、红冲还是补偿。

## 16. 支付管理与支付完成附加动作

支付单必须带原始业务 payload，支付管理不能事后猜。

支付单来源需要包含：

- 来源业务对象。
- 业务类型。
- 付款方/收款方。
- 收款账户。
- 金额拆分。
- 交易信息。
- 关联合同、项目、车辆、司机、供应商、货主、期间。
- 触发状态。
- 失败重试和反向处理。

支付完成后常常有附加动作：

- 回写来源单据支付状态。
- 输出供应商工资明细。
- 推送外部系统。
- 解冻/释放资金。
- 更新上次发放时间。
- 生成流水、通知、报表统计。

这些附加动作要有幂等、重试、失败可见、人工恢复。不能只写“支付成功后系统自动处理”。

## 17. 待确认要尽量关闭

用户强调：待确认要尽量关闭，除非用户明确表示无法确定。

这意味着：

- 有业务锚点时，用锚点关闭。
- 有既有系统行为时，用现状关闭。
- 有低风险行业惯例时，给推荐默认值。
- 只有高风险且无依据时，才保留待确认。

推荐写法：

```text
推荐默认：工资单期间由“上次工资发放月份 + 1 个自然月”生成，创建弹窗不可改。
依据：该字段用于防止重复发放和保持连续性。
需确认：作废已支付工资单时，进度字段回滚还是保留并走调整单。
```

## 18. 主文件要先让人看懂产品

00 主文件不能只放规则索引。它要先帮助人理解：

- 有哪些端。
- 有哪些模块。
- 每个模块有哪些页面。
- 页面里有哪些功能点。
- 业务主流程怎么跑。
- 哪些流程并行。
- 哪些对象状态变化驱动下游。

所以主文件需要：

- 端 / 模块 / 页面 / 功能点思维导图。
- 业务流程图谱。
- 每条并行流程的图。
- 关键对象状态图。
- 汇合点和回写点说明。

这样用户一眼知道功能在哪里，开发也知道模块怎么切。

## 19. 模块文档的阅读顺序

用户希望模块文档更像人看的文档，而不是一堆表。

推荐结构：

1. 模块是干嘛的。
2. 主流程是什么。
3. 页面长什么样。
4. 关键操作有哪些。
5. 风险和异常。
6. 页面里的落地细节：字段、数据来源、状态回写、权限、幂等、验收。
7. 无页面参与的后台自动流程，也放在所属模块里。

关键点：字段、数据来源、状态、权限不要拆成互相跳转的独立大章，而是放在页面、操作、后台流程的上下文里。

## 20. 通用化：不是财务专用

这些思维方式未来也适用于非财务业务。

| 财务里的问题 | 非财务里的对应问题 |
|---|---|
| 余额/授信/冻结 | 库存/额度/权限/容量 |
| 支付完成回写 | 任务完成回写、资格更新、进度推进 |
| 对账明细占用 | 工单、订单、库存明细被批次占用 |
| 发票子对象 | 附件、证明、审批意见、外部回执 |
| 分润资金瀑布 | 积分、优惠、会员权益、资源分配瀑布 |
| 异常补车辆 | 异常补用户、补组织、补映射、补外部 ID |
| 上次工资发放时间 | 上次发券时间、上次结算期、已处理至批次 |

所以 skill 不应写成“财务规则大全”，而应写成“业务闭环思维工具”。

## 21. 未来使用 skill 时的讨论顺序

建议 AI 和用户讨论时按这个顺序：

1. 先复述业务理解：基于用户上下文，不套模板。
2. 标出已确认、推断、冲突、缺口。
3. 做深度业务理解包：调研、别扭流程、小流程、异常反向。
4. 判断流程形态：单线、并行、主子、状态机、事件驱动、周期性。
5. 画主流程和对象状态。
6. 用观察视角发现小流程。
7. 对关键资源做闭环：来源、占用、组合、消耗、释放、角色视图。
8. 对关键字段做控制闭环：谁读、怎么默认、何时推进、失败不推进、反向怎么处理。
9. 对页面做任务闭环：为什么存在、先看什么、在哪操作、失败怎么恢复。
10. 最后才写模块 PRD。

## 22. 最重要的一句话

这套 skill 的核心不是“帮 AI 写更完整的 PRD”，而是：

```text
让 AI 在写 PRD 前，先真正理解业务、发现业务、挑战业务，再把业务落成可实现的产品需求。
```

完整不是堆字段，严谨不是堆表格。真正有用的 PRD，是业务、流程、页面、数据、状态、异常和验收都闭环，而且人能看懂。
